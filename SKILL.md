# 药厂多部门排班调度

把药厂项目从原料采购到成品出货的全流程排班，变成一个可串联、可追踪的系统。

## 设计理念

> 一个药品项目涉及 4 个核心部门，它们之间有严格的先后依赖：
>
> ```
> 采购下单 → 原料到货 → QC来料检验 → 车间生产 → QC成品检验 → 放行 → 销售发货
> ```
>
> 任何一个环节延迟，后面全部跟着延迟。本 skill 的核心价值是：
> **排一个部门的班时，自动考虑上游什么时候交付，下游什么时候需要。**

## 核心模型：资源池（参考 Smart-QC / Binocs）

```
┌─────────────────────────────────────────────────────────┐
│                      资 源 池                            │
│  ┌──────────┐  ┌──────────┐  ┌──────────┐              │
│  │ 人员池   │  │ 仪器池   │  │ 房间池   │              │
│  │ 张三 QC  │  │ HPLC-1   │  │ 天平室   │              │
│  │ 李四 QC  │  │ UV-1     │  │ 微生物室 │              │
│  │ 王五 车间│  │ 溶出仪   │  │ 理化室   │              │
│  └──────────┘  └──────────┘  └──────────┘              │
│       ↕              ↕              ↕                   │
│  ┌──────────────────────────────────────┐              │
│  │         时间槽（每天上午/下午）        │              │
│  │  07-01上午: 张三+HPLC-1 → 含量测定    │              │
│  │  07-01下午: 张三+HPLC-1 → 有关物质    │              │
│  │  07-01上午: 李四+UV-1   → 鉴别        │              │
│  └──────────────────────────────────────┘              │
│       ↕                                                 │
│  ┌──────────────────────────────────────┐              │
│  │         约束引擎                      │              │
│  │  硬约束: 仪器同时只能1人/校准过期不可用│              │
│  │  软约束: 人员偏好/缓冲时间/优先级     │              │
│  └──────────────────────────────────────┘              │
└─────────────────────────────────────────────────────────┘
```

### 资源池运作方式

1. **注册资源**：所有人员、仪器、房间注册到资源池
2. **时间切片**：每个工作日切为上午(8:00-12:00)、下午(13:00-17:00) 两个时段
3. **自动分配**：任务按优先级排序，逐个塞入可用时间槽
4. **冲突检测**：同一资源同一时段只能分配一个任务
5. **使用率统计**：自动计算人员负荷、仪器使用率

### 硬约束（不可违反）
- 仪器同一时段只能一个人用（除非有多台）
- 校准过期的仪器不可排班
- 仪器状态异常不可排班
- 非工作日不排班（自动排除周末+法定假日）
- 上游部门未放行，下游部门不可开始

### 软约束（尽量满足）
- 优先级高的任务优先排
- 仪器需求多的任务优先排（资源竞争大）
- 人员偏好（如上午做精密检验）
- 留 10-20% 缓冲应对异常

## 资源池模型（已实现）

系统采用**资源池模型**（参考 Smart-QC/Binocs）：
```
资源池 = {人员池, 仪器池, 房间池}
每个任务 = 需要 X 个资源 × Y 小时
排班 = 把任务塞进资源池的时间槽，自动避免冲突
```

核心组件（`shared/common.py`）：
- `ResourcePool` — 统一管理人员/仪器/房间的时间槽，自动检测冲突和校准过期
- `SchedulingEngine` — 约束驱动排程引擎，支持批量自动排程和甘特图输出
- 硬约束：资源可用性、校准有效期、依赖关系
- 软约束：人员偏好、缓冲时间、优先级

## 脚手架使用

`scaffold_pharma.py` 一键生成排班目录。路径：`~/.hermes/skills/pharma-schedule/scripts/scaffold_pharma.py`

依赖 `shared/common.py`（部门定义、依赖引擎、时间推导、风险检测），Python 3.10+。

```bash
python3 ~/.hermes/skills/pharma-schedule/scripts/scaffold_pharma.py /目标目录 \
  --project-name "XX片剂批生产" \
  --drug-name "XX片" \
  --batch-no "20260701-01" \
  --start-date 2026-07-01 \
  --deadline 2026-08-15 \
  --departments "procurement,qc_incoming,workshop,qc_finished,sales" \
  --dept-durations '{"procurement":7,"qc_incoming":3,"workshop":5,"qc_finished":5,"sales":3}' \
  --personnel-json '[{"name":"张三","department":"qc_incoming","level":"QC工程师"}]' \
  --instruments-json '[{"name":"HPLC-1","model":"Agilent","location":"qc_finished","cal_due":"2026-12-31"}]' \
  --test-items-json '[{"name":"含量测定","department":"qc_finished","assignee":"张三","duration":"3天"}]'
```

## 执行步骤

### 第一步：收集项目总信息

| 字段 | 说明 | 示例 |
|------|------|------|
| 项目名称 | 药品名+批类型 | XX片剂 批生产 |
| 药品名称 | 通用名 | XX片 |
| 批号 | 批号 | 20260701-01 |
| 批量 | 生产批量 | 10万片 |
| 项目类型 | 新品种/常规生产/验证/其他 | 常规生产 |
| 质量标准 | 依据标准 | 中国药典2025版 |
| 项目启动日期 | 整体启动 | 2026-07-01 |
| 交货截止日 | 销售需要的交货时间 | 2026-08-15 |
| 特殊要求 | 加急、特殊工艺等 | 客户催单，尽快 |

### 第二步：确认参与部门

根据项目类型自动判断需要哪些部门：

| 项目类型 | 必须参与 | 可选参与 |
|----------|----------|----------|
| 常规生产 | 采购 + QC来料 + 车间 + QC成品 + 销售 | — |
| 新品种 | 采购 + QC全项 + 车间(工艺验证) + 销售 | 研发 |
| 验证批 | QC + 车间 | 采购 |
| 来料检验 | 采购 + QC来料 | — |
| 稳定性考察 | QC | 销售 |

### 第三步：确定关键时间节点（反向推导）

从交货截止日**反向**推导各部门的最晚开始时间：

```
交货日 T
  ← 销售发货准备: 2天
  ← QC成品检验放行: 5天（含微生物）
  ← 车间生产: 5天（含包装）
  ← QC来料检验: 3天
  ← 原料到货: 采购周期（7-30天视品种）
  ← 采购下单: 1天
```

系统自动算出每个部门的**最晚启动日**和**截止日**，标注是否赶工。

### 第四步：逐部门排班

按依赖顺序，依次为每个部门生成排班：

1. **采购部排班** → 下单、跟催、到货验收
2. **QC来料排班** → 原辅料检验、出具报告
3. **车间排班** → 称量、配制、压片、包衣、包装
4. **QC成品排班** → 成品全项检验、出具报告
5. **销售排班** → 发货计划、物流安排

每个部门的排班格式参考对应子 skill：
- `qc` → QC 排班（来料/成品通用）
- `workshop` → 车间生产排班
- `sales` → 销售发货排班
- `procurement` → 采购排班

### 第五步：生成串联看板

输出一份**跨部门甘特图**和**关键路径**：

```markdown
## 项目时间线

| 阶段 | 部门 | 开始 | 截止 | 负责人 | 前置依赖 | 状态 |
|------|------|------|------|--------|----------|------|
| 原料采购 | 采购部 | 07-01 | 07-08 | 赵六 | — | ⬜ |
| 包材采购 | 采购部 | 07-01 | 07-10 | 赵六 | — | ⬜ |
| 原料检验 | QC来料 | 07-09 | 07-11 | 张三 | 原料到货 | ⬜ |
| 批生产 | 车间 | 07-14 | 07-18 | 生产主管 | 原料放行+包材到货 | ⬜ |
| 成品检验 | QC成品 | 07-19 | 07-23 | 李四 | 车间完工 | ⬜ |
| 放行审核 | QA | 07-24 | 07-25 | QA经理 | 成品检验合格 | ⬜ |
| 发货 | 销售部 | 07-26 | 07-28 | 销售 | 放行 | ⬜ |

⚠️ 关键路径: 采购 → QC来料 → 车间 → QC成品 → 放行 → 发货
⚠️ 总工期: 28天 | 缓冲: 18天（截止08-15）
```

### 第六步：风险预警

自动检测：

- **时间冲突**：上游未完成，下游已到开始时间
- **仪器冲突**：多部门同时需要同一台仪器
- **人员冲突**：一人同时排了不同部门的任务
- **校准过期**：排班期内仪器校准到期
- **节假日**：排班落在法定假日
- **赶工风险**：某部门排班过于紧凑，无缓冲

## 单部门排班模式

当用户只需要排 QC 一个部门的班时（不做全流程），跳过"确认参与部门"和"反向推导"步骤，直接进入 QC 排班：
- 触发词：QC排班、检验排期、仪器排期、新项目检验安排、人员分配
- 不需要采购/车间/销售信息
- 只输出：检验项目清单、人员排班表、仪器排期表、每日工作看板
- 检验项目×仪器×耗时参考 `references/qc-domain-knowledge.md`

## Pitfalls

### 1. Instrument 对象是 dataclass 不是 dict

`scaffold_pharma.py` 中 `instruments` 列表的元素是 `Instrument` dataclass 对象（定义在 `shared/common.py`），**不能用 `.get()` 访问**。必须用 `getattr(obj, "field", default)` 或直接 `obj.field`。

```python
# ❌ 错误
inst.get("location")
inst["name"]

# ✅ 正确
getattr(inst, "location", None)
inst.name
```

此 bug 在首次测试中出现并修复。如果修改 scaffold 脚本新增对 instrument 字段的访问，务必保持一致。

### 2. 反向推导可能产生紧张排期

从截止日反向推导时，如果总工期 ≈ 各部门工期之和，每个部门的缓冲只有 0-1 天。脚本会自动标注风险，但要主动告知用户：
- 建议至少 10-20% 缓冲
- 微生物/无菌培养周期是刚性的，不能压缩
- 如果缓冲不足，建议提前启动日期或推迟交货日

### 3. 节假日处理

`shared/common.py` 中硬编码了 2026 年中国法定节假日。如果排班跨年或用到下一年，需要更新 `HOLIDAYS_2026` 列表。节假日数据需要每年维护。

## 注意

- **不要跳过依赖**：上游未放行，下游不能开始（GMP 强制）
- **反向推导**：从交货日倒推，不是从启动日正推
- **每个部门独立排班**：各部门排班细节参考对应子 skill
- **仪器共享**：QC 和车间可能共用某些设备（如天平、pH 计），需交叉排期
- **灵活增减部门**：用户说不需要销售就去掉，说需要研发就加上
- **用户偏好**：用户是药厂 QC 背景，熟悉 GMP 流程，不需要解释基础概念，直接执行
